# ADR-002 — v1.1: usuários reais, multi-tenancy e monetização

**Status:** Proposto (aguardando validação do usuário antes de qualquer código)
**Data:** 2026-07-24

## Contexto

A Plataforma MCP Brasil hoje não tem conceito de usuário. Tudo que parece
"por usuário" — a Carteira (`carteira_posicoes`, `carteira_snapshots`,
migration 005) — na verdade é escopado por um **`session_id` gerado no
navegador** (`crypto.randomUUID()`, salvo em `localStorage`,
`frontend/lib/carteira.ts:15`). Não há login, não há `auth.users`, não há
Row Level Security em nenhuma tabela.

Isso funciona para um projeto pessoal de demonstração, mas quebra assim que
existe qualquer noção de "conta paga", "meus dados privados" ou "meu
histórico de chat":
- **Um `session_id` não sobrevive** a trocar de navegador, limpar cache, ou
  trocar de dispositivo — a carteira "some".
- **Não há isolamento real**: quem descobrir/adivinhar o `session_id` de
  outra pessoa acessa a carteira dela (as rotas `GET /carteira/*` só exigem
  o `session_id` como query param — ver `backend/routes/carteira.py`).
- **Não há como cobrar** de ninguém por nada — não existe identidade estável
  pra associar a um plano/assinatura.
- **O Copiloto não pode ter memória** entre sessões sem saber "de quem" é a
  conversa.

Nesta sessão, o usuário definiu a direção da v1.1 (sem prazo definido):
1. **Foco principal: monetização / usuários reais.**
2. **Carteira**: manter entrada manual **+** importação automática (B3 ou
   corretora, método ainda em aberto) **+** alertas/recomendações baseados
   nas métricas de risco já calculadas (Sharpe, Sortino, Calmar, Drawdown).
3. **Copiloto**: quer **memória entre sessões** (não pediu web search de
   volta).
4. Sem prazo — evolução orgânica.

Todos os itens 2 e 3 **dependem** de identidade de usuário real (item raiz).
Esse é o motivo deste ADR: alinhar a fundação antes de qualquer feature em
cima dela.

## Decisão proposta

Adotar **Supabase Auth** como camada de autenticação, pelos motivos:
- O banco já é Supabase — zero infraestrutura nova, zero custo adicional no
  free tier (Supabase Auth é incluído).
- Dá **Row Level Security (RLS)** nativa via `auth.uid()` no Postgres —
  isolamento de dados por usuário embutido no banco, não reimplementado em
  código de aplicação (mais seguro que checar `session_id` em cada rota).
- Suporta e-mail/senha e OAuth (Google) sem servidor adicional.
- O frontend já é Next.js — `@supabase/ssr` cobre sessão client+server sem
  gambiarra.

### Sequência de fases (cada uma é um PR/grupo de PRs independente)

#### Fase 0 — Fundação de identidade (bloqueia tudo abaixo)
- Habilitar Supabase Auth (e-mail/senha inicialmente; OAuth Google depois).
- Migration: `carteira_posicoes.session_id` e `carteira_snapshots.session_id`
  passam a referenciar `auth.users(id)` — **renomear pra `user_id`**
  mantendo o tipo compatível (é a mesma "chave de tenant" que já existe,
  só troca a origem: de UUID client-side pra UUID de usuário autenticado).
- Habilitar RLS nas duas tabelas: policy `user_id = auth.uid()`.
- Frontend: tela de login/cadastro, `getSessionId()` vira `getUserId()` lendo
  da sessão Supabase Auth em vez de `localStorage`.
- Backend: rotas de carteira trocam `session_id: str = Query(...)` por
  autenticação via JWT do Supabase (validar o token, extrair `user_id`) —
  fecha também o buraco de "adivinhar session_id de outro".
- **Migração de dados existentes**: como não há usuários reais hoje (só
  sessões anônimas de teste), não há dado de produção real a migrar —
  simplifica a Fase 0 (não precisa de estratégia de merge/dedup).

#### Fase 1 — Memória do Copiloto (depende da Fase 0)
- Nova tabela `copilot_conversas` (ou `copilot_mensagens`): `user_id`,
  `session_conversa_id`, `role`, `conteudo`, `criado_em`. RLS por `user_id`.
- `native_agent.perguntar()` ganha parâmetro opcional de histórico — busca
  as últimas N mensagens do usuário antes de montar o `messages=[...]` do
  `tool_runner`.
- Trade-off a decidir na hora: janela de contexto (últimas N mensagens vs.
  resumo) e custo (mais histórico = mais tokens de input por pergunta).

#### Fase 2 — Monetização (depende da Fase 0)
- Modelo de planos (ex.: Free = 1 carteira manual, Pro = importação
  automática + alertas + Copiloto sem limite de perguntas/dia).
- Integração de cobrança (Stripe é o padrão de mercado; BR também tem
  Pagar.me/Mercado Pago se quiser PIX nativo — a decidir).
- Tabela `assinaturas` ou uso do próprio Supabase + webhook do provedor de
  pagamento pra atualizar o plano do usuário.
- Rate limiting por plano nas rotas (especialmente `/copilot/*`, que
  consome API paga da Anthropic por pergunta).

#### Fase 3 — Carteira automática + alertas (depende da Fase 0, pode rodar em paralelo à Fase 2)
- **Importação**: método ainda em aberto (usuário não decidiu). Opção mais
  simples pra começar: upload de nota de corretagem/extrato B3 (PDF/Excel)
  com parsing — não depende de homologação com corretora nenhuma. Integração
  direta com API de corretora é opção futura, mas exige parceria/homologação
  por corretora (não há uma API B3 pública de posições de cliente).
- **Alertas**: usar as métricas já calculadas (`carteira/metricas.py`) pra
  gerar notificação quando, por ex., Drawdown ultrapassa um limite ou
  concentração num ticker passa de X% — canal de entrega a definir
  (e-mail via Supabase, ou só in-app).

## Consequências

- **Toda rota de carteira muda de contrato** (query param `session_id` →
  JWT no header `Authorization`) — é breaking change de API, mas como não
  há usuários reais hoje, o custo de migração é baixo agora e só cresce
  quanto mais se adia.
- **Infra free tier vai apertar mais rápido**: usuários reais geram mais
  tráfego que o teste solo de hoje — o Render free tier (single worker,
  cold start) e o Supabase free tier (500MB) já mostraram limite nesta
  sessão só com QA automatizado. Monetização real provavelmente **exige**
  upgrade de infra antes de escalar (já sinalizado no README como decisão
  de negócio em aberto).
- **Custo do Copilot com memória sobe**: mais contexto por pergunta = mais
  tokens de input. Vale medir antes de liberar sem limite no plano Free.

## Alternativas consideradas

- **Auth0 / Clerk / NextAuth com provider próprio**: mais flexível, mas
  adiciona um serviço externo novo (custo, mais uma conta a gerenciar) sem
  ganho claro sobre o Supabase Auth que já vem "de graça" com o banco atual.
  Descartado por ora — reconsiderar só se o Supabase Auth mostrar limitação
  concreta.
- **Manter `session_id` e só adicionar "vincular a e-mail" por cima**: mais
  rápido de implementar, mas não resolve o problema de segurança (RLS) nem
  dá uma base sólida pra cobrança. Rejeitado.
